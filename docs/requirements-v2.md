# 听课助手（class_copilot）— 需求文档

> 版本：v2.0
> 日期：2026-03-20
> 状态：需求对齐中

---

## 1. 项目概述

### 1.1 项目背景

大学课堂中，教师经常在课上提出问题，学生积极回答可以获得**课堂表现分**。然而：

- 课程内容涉及面广，学生不一定能快速组织出完整、得体的回答
- 课堂节奏快，思考时间有限（老师提问后约 10 秒内需要举手）
- 回答需要符合课程语境，措辞得当

本项目旨在构建一个**实时听课辅助工具**，帮助学生在课堂上快速获取高质量的答题参考，提升课堂表现。同时支持课堂实时问答，帮助学生理解课堂内容。

### 1.2 目标用户

- 大学生（主要场景：思政课等中文授课课程，同时支持英文授课课程）
- 可扩展至其他课程类型

### 1.3 使用环境

| 项目 | 说明 |
|------|------|
| 设备 | 笔记本电脑（Windows） |
| 电源 | 可能无外接电源，需低功耗 |
| 网络 | 校园网（稳定） |
| 麦克风 | 笔记本内置麦克风 |
| 课堂时长 | 约 90 分钟 |
| 教室环境 | 安静，无明显噪音干扰 |
| 课堂人员 | 一般是老师在说话；偶尔有同学做 Presentation |

## 2. 核心功能需求

### 2.1 实时语音转写

#### 2.1.1 基本功能

- 实时采集课堂音频，将语音转为文字
- 支持连续 90 分钟稳定转写
- 转写延迟目标 ≤ 3 秒
- 识别准确率 ≥ 90%
- 录音手动开始/停止

#### 2.1.2 语言支持

- 支持中文授课
- 支持英文授课
- 自动识别中英混合内容

#### 2.1.3 转写显示

- 实时滚动显示转写文本，自动滚动到最新内容
- 中间结果（未确认）用灰色显示，最终结果用白色显示
- 中间结果在最终结果到达后自动替换
- 不同说话人用不同颜色区分（教师、学生等）
- 显示说话人标签（如"教师""发言"等）
- 支持一键复制当前转写文本到剪贴板

#### 2.1.4 英文翻译与双语展示

- 英文授课时，可选开启自动翻译为中文
- 可选开启中英双语展示模式：同时显示英文原文和中文翻译
- 翻译失败时不影响正常转写显示

#### 2.1.5 课程专业术语

- 支持用户输入课程相关的专业术语（热词）
- 用于提升特定领域的识别准确率

---

### 2.2 说话人识别

#### 2.2.1 说话人分离

- 自动区分不同说话人
- 支持用户手动标记某个说话人为"教师"
- 标记后，该说话人在转写中以教师身份高亮显示

#### 2.2.2 声纹管理

- 支持注册教师声纹，下次上课自动识别
- 声纹按「课程 + 教师名称」关联管理
- 支持查看、删除已注册的声纹
- 声纹特征云端存储，映射关系本地保存

---

### 2.3 问题检测

#### 2.3.1 自动检测

- 从转写文本中自动识别教师提出的问题
- 基于疑问句式、关键词和 LLM 辅助判断
- 检测置信度阈值筛选，避免误报
- 短时间内避免重复检测相同问题（相似度 ≥ 80% 视为重复）
- 两次检测之间有最小冷却间隔（避免频繁触发）

#### 2.3.2 手动触发

- 支持快捷键手动触发问题检测，作为自动检测的补充
- 手动触发时使用最近若干条转写片段作为输入

#### 2.3.3 LLM 输入过滤

- 默认仅将教师发言送入 LLM 进行问题检测
- 可切换为将所有人的发言送入（适用于同学 Presentation 等场景）
- 支持快捷键切换过滤模式

#### 2.3.4 问题提醒

- 检测到问题时弹出 Windows 系统通知
- 通知无声音（避免课堂暴露）
- 通知显示问题概要

---

### 2.4 智能答案生成

#### 2.4.1 答案生成

- 检测到问题后自动生成参考答案
- 也支持手动触发生成（即使 LLM 未判定为问题，也可强制生成）
- 基于课堂转写上下文和课程名称生成答案
- 风格要求：**口语化**，像学生口头回答，自然流畅
- 答案语言跟随授课语言；英文授课时提供英文+中文双语答案
- 答案生成延迟目标 ≤ 3 秒

#### 2.4.2 双版本答案

- 每次生成**简洁版**和**展开版**两个答案
- 简洁版：2-3 句话，快速响应
- 展开版：5-8 句话，详细解释
- 两个版本并行生成
- 用户可在设置中独立开关简洁版/展开版

#### 2.4.3 答案展示

- 每个检测到的问题以卡片形式展示
- 卡片显示来源图标：自动检测 🔍 / 手动触发 ✋ / 强制生成 💬 / 精修补检 🔄
- 默认显示简洁版，提供切换按钮查看展开版
- 支持一键复制当前答案到剪贴板
- 答案列表按时间倒序排列（最新在前）
- 生成中显示"正在生成答案…"占位

---

### 2.5 主动提问（课堂实时问答）

#### 2.5.1 基本功能

- 用户可以主动输入问题，基于当前课堂上下文获取 AI 回答
- 使用场景：课堂上遇到不理解的内容，实时向 AI 提问
- 自动携带当前课堂转写记录作为上下文

#### 2.5.2 交互方式

- 提供输入框，支持回车键或按钮提交
- 用户问题显示为"🙋 你: [问题]"
- AI 回答显示为"🤖 AI: [回答]"
- 问答按时间顺序展示
- 支持快捷键快速呼出提问界面

#### 2.5.3 回答质量

- 主动提问使用更强大的模型（相比自动答案），确保回答深度
- 支持用户选择模型（快速/高质量）
- 支持"深度思考"模式开关（更慢但更准确）

#### 2.5.4 Markdown 渲染

- AI 回答支持 Markdown 格式渲染
- 包括：代码块（语法高亮）、行内代码、标题、粗体/斜体、有序/无序列表等

### 2.6 高精度精修转写（可选功能）

> 本节所有功能均为**可选**，用户可在设置中自由开关。精修层的任何故障不影响实时转写和问题检测等核心功能。
>
> **核心原则：精修层是增强，不是替代。即使精修层完全不可用，系统行为与关闭精修时完全一致。**

#### 2.6.1 背景与动机

- 实时 ASR 在教师语速快、口音重、专业术语密集时，准确率明显下降
- 转写错误导致**问题检测漏检**或**答案质量下降**
- 课后回顾历史记录时，转写质量不够理想，可读性差
- 录音文件已在本地保存（MP3 格式），但未被二次利用

#### 2.6.2 功能定位

- 在实时 ASR（低延迟、中精度）基础上，引入高精度非实时 ASR 作为**增强层**
- 可接受分钟级延迟，换取显著更高的转写精度
- 特别适用于：专业术语密集段落、中英混合内容、教师口述的定义/公式

| 层级 | 角色 | 延迟 | 精度 |
|------|------|------|------|
| **实时层**（现有） | 保障课堂实时交互：问题检测、答案生成、主动提问 | ≤ 3 秒 | 中 |
| **精修层**（新增） | 修正转写文本，提升历史记录质量，补检遗漏问题 | 1~3 分钟 | 高 |

#### 2.6.3 精修模型

- 使用阿里百炼 **qwen3-asr-flash**（文件转写版本）
- 复用现有 DashScope API Key，无需额外配置
- 支持中文、英文、中英混合
- 支持课程热词提升专业术语识别率

#### 2.6.4 触发策略（用户可选）

##### 课后自动批量精修（默认策略）

- 用户点击"停止监听"结束课堂后，**自动开始**对本次课堂录音进行高精度转写
- 无需用户额外操作
- 后台批量处理，用户可正常使用其他功能（浏览历史、查看答案等）
- 进度在状态区域低调显示："精修中 (35%)"
- 完成后状态变为："精修完成 ✓"，持续显示 5 秒后消失
- 优点：不消耗课中带宽和 API 额度，课堂体验零干扰
- 缺点：课中无法享受修正带来的增强

##### 课中定时分段精修（可选）

- 每隔固定间隔（建议 3~5 分钟，可配置），将该时段的录音片段提交高精度转写
- 课堂进行中持续滚动触发
- 后台静默提交，返回修正结果后替换对应时段的转写文本
- 典型延迟：提交后 1~3 分钟返回结果
- 优点：课中即可享受精修增强（补检漏掉的问题、优化答案上下文）
- 缺点：额外网络带宽消耗，实现复杂度更高

##### 手动触发精修（始终可用）

- 用户主动点击"立即修正"按钮或使用快捷键
- 可选修正范围：全部 / 最近 N 分钟
- 适用于对成本敏感的用户，或只想对关键片段精修

##### 策略组合

- 三种策略可任意组合
- 用户可选一个自动策略（课后 或 课中定时），手动触发始终可用
- 课中定时与课后批量可同时开启

#### 2.6.5 精修结果的用户可见行为

##### 课堂实时阶段

- **实时转写区域行为与关闭精修时完全一致**：文字逐句出现，中间结果灰色、最终结果白色
- 用户无需关心背后是否有精修在进行
- 精修完成的片段**原地静默替换**实时转写文本（文字内容更新，位置不变）
- **不弹出通知、不打断阅读流、不自动滚动**
- 如果用户正在阅读某段文本且该段刚好被修正，采用**渐变过渡**（如 0.3s 淡入替换），避免突兀跳变

##### 修正状态展示

| 状态 | 展示方式 |
|------|---------|
| 仅有实时转写（未修正） | 无特殊标记（与关闭精修时一致） |
| 精修进行中 | 该段落右侧显示微小的旋转图标（用户主动查看该区域时可见） |
| 精修完成 | 微小 ✓ 角标，鼠标悬停显示"已修正" |
| 精修失败 | 无标记，保留实时版本，用户无感知 |

##### 修正前后对比（可选）

- 用户可在历史记录中**主动开启"对比模式"**，查看实时版 vs 精修版的差异（类似 diff 视图）
- 默认关闭，不主动展示差异
- 数据库中保留实时版和精修版两份文本

##### 课堂结束后

- 历史记录中展示的是**最终版本**（精修版优先，未修正部分保留实时版）
- 精修尚未完成时标注"精修进行中…"，完成后标注消失

#### 2.6.6 对其他功能的联动（均为可选）

##### 问题检测

- 实时检测**保持不变**：继续基于实时转写文本触发，保证 ≤ 6s 端到端响应不受影响
- 精修后**自动二次补检**（可关闭）：精修文本到达后，对该段重新检测是否有遗漏的问题
- 补检到的新问题追加到问题列表，标记来源为"精修补检 🔄"
- 已在实时阶段检测到的问题（相似度匹配）不重复添加，但可用精修文本更新问题描述
- 课中补检到的遗漏问题**仍正常通知**（可能还来得及回答）
- 课后补检到的遗漏问题**仅存入记录，不通知**

##### 答案生成

- 实时答案**保持不变**：实时检测到问题后立即基于实时转写上下文生成
- 精修后**自动更新答案**（可关闭）：若精修文本与实时版有实质性差异（非标点/语气词级别），自动用精修文本重新生成答案，替换原答案
- 答案卡片上标注"已基于精修文本更新"（小字灰色提示）
- 课后补检到的新问题生成答案存入记录，供历史回顾
- 已被用户复制走的答案不受影响

##### 主动提问

- 上下文**始终使用当前最优版本**（默认开启）：已精修的用精修版，未精修的用实时版
- 这是静默优化，用户无需感知

##### 历史记录与导出

- 历史记录展示**最优版本**（精修版 > 实时版）（默认开启）
- 导出**默认使用精修版**（如可用），未修正部分用实时版补全
- 导出设置提供选项：☑ 使用高精度修正版本（默认）/ ☐ 同时导出原始版本 / ☐ 导出对比标注版本

##### 说话人识别

- 高精度 ASR 通常有更准确的说话人分离能力
- 精修后的说话人标签**以高精度版本为准**，覆盖实时版标签
- 声纹标记关系（哪个 speaker_label = 教师）仍以用户手动标记或声纹匹配为准
- 通过**时间戳对齐 + 文本相似度匹配**，将高精度说话人标签映射到已有角色标记
- 高精度版分离出实时版未识别的新说话人，标记为 UNKNOWN
- 存在冲突时以高精度版为准

#### 2.6.7 异常处理

##### ASR 失败

| 场景 | 处理 |
|------|------|
| API 调用失败（网络错误、超时） | 自动重试最多 3 次，间隔递增（10s / 30s / 60s），全部失败后放弃该片段，保留实时版本 |
| API 返回空结果或异常结果 | 丢弃该结果，保留实时版本 |
| API 密钥失效或额度耗尽 | 停止修正，状态区域提示"精修服务不可用：[原因]"，实时功能不受影响 |

##### 音频文件问题

| 场景 | 处理 |
|------|------|
| 录音文件损坏 | 跳过损坏片段，尝试修正其他片段，标注"部分片段因音频损坏跳过" |
| 录音文件缺失 | 提示"录音文件缺失，无法精修"，实时转写保持不变 |
| 磁盘空间不足导致录音截断 | 修正已有部分，缺失部分保留实时版本 |

##### 网络中断

| 场景 | 处理 |
|------|------|
| 课后修正时网络中断 | 暂停修正，记录进度，网络恢复后从断点继续 |
| 课中修正时网络中断 | 暂停精修提交，实时 ASR 断网逻辑不受影响，网络恢复后从新片段继续 |
| 长时间离线 | 修正任务保留为"待处理"，下次联网时可手动或自动恢复 |

##### 成本保护

| 场景 | 处理 |
|------|------|
| 单次课堂音频超过上限 | 修正到上限时暂停，弹窗提示"已达修正上限"，用户选择继续或停止 |
| 异常长时间录音（忘记关闭） | 超过 3 小时未操作时自动暂停精修任务（不暂停录音） |
| 用户想查看已用额度 | 设置页面提供"本月精修用量"统计（累计音频时长） |

##### 应用崩溃/意外退出

| 场景 | 处理 |
|------|------|
| 精修过程中应用崩溃 | 下次启动时检查未完成任务，提示"上次精修未完成，是否继续？" |
| 精修完成但结果未写入数据库 | 原子性保证：精修结果写入与状态更新在同一事务中，崩溃后回退到实时版本，可重新精修 |

### 2.7 课堂记录与存储

#### 2.7.1 自动保存

- 每节课完整录音自动保存为音频文件
- 完整转写文本（含说话人标签和时间信息）自动保存
- 检测到的问题和生成的答案自动保存
- 用户主动提问的问答记录自动保存
- 数据永久保留，不自动清理

#### 2.7.2 录音文件

- 录音保存为压缩音频格式
- 按日期和课程名归档存储
- 录音中定期刷新到磁盘，防止崩溃丢失

#### 2.7.3 会话续记

- 支持从历史会话继续录制（续记）
- 续记时加载之前的转写记录作为历史上下文
- 历史内容与新内容有明确视觉分隔
- 续记时 LLM 可利用之前的转写上下文

##### 续记与精修的交互

- 续记时加载的历史上下文**使用已有最优版本**：
  - 前段已精修完毕 → 加载精修版
  - 前段正在精修中 → 加载实时版（不等待精修完成）
  - 前段精修失败 → 加载实时版
- 续记过程中，前段精修完成后，**不会动态替换已加载的历史上下文**（避免 LLM 上下文突变）
- 续记结束后，整体历史记录自动使用最优版本
- 每段录音独立进行精修，互不影响
- 一个会话中可能有多个录音文件，每个文件精修状态独立
- 历史记录中统一展示，不暴露录音分段细节

##### LLM 上下文版本选择

| 场景 | 上下文版本 | 理由 |
|------|-----------|------|
| 实时问题检测 | 实时版（当前片段）+ 最优版（历史片段） | 当前正在发生的内容只有实时版 |
| 实时答案生成 | 同上 | 保持一致 |
| 主动提问 | 所有片段的最优版本 | 无极端时效性要求，用最好的上下文 |
| 精修后二次检测 | 精修版（本段）+ 最优版（前后段） | 二次检测本身就是为了用更好的文本 |

#### 2.7.4 历史记录浏览

- 列表显示所有历史会话，按日期倒序排列
- 显示格式：日期 + 课程名
- 标记异常会话状态（如意外中断）
- 标记精修状态（精修中 / 已完成）
- 点击查看完整会话详情：转写、问答记录

#### 2.7.5 导出

- 支持导出为 Markdown 格式文件
- 导出内容包含完整转写和所有问答记录
- 精修版文本优先（如可用），未修正部分用实时版补全

#### 2.7.6 删除

- 支持删除历史会话（含音频、转写、问答记录）
- 删除前弹窗确认

## 3. 用户界面需求

### 3.1 整体形态

- **Web 应用**，本地启动后端服务，浏览器访问
- 后端运行在本地（localhost），前端通过浏览器打开
- 启动应用后自动打开默认浏览器
- 深色主题（课堂上更不显眼）
- 响应式布局，适配不同窗口大小
- 支持将浏览器窗口缩小到侧边，不影响功能

### 3.2 主界面布局

#### 3.2.1 顶部导航栏

- 应用名称 / Logo
- 课程名输入框（下拉框，记忆历史课程名）
- 开始/停止录音按钮（醒目的状态切换）
- 连接状态指示（实时 ASR 状态、精修状态）
- 设置入口（齿轮图标）

#### 3.2.2 状态栏

- 显示当前状态：就绪、正在监听、已停止、正在重连等
- 显示错误/提示信息（Toast 通知）
- 精修进度显示（如已开启）：「精修中 3/10 段」

#### 3.2.3 主内容区（选项卡切换）

- **转写**：实时转写文本显示区，实时滚动
- **回答**：检测到的问题和答案卡片列表
- **提问**：主动问答输入和对话展示区
- **历史**：历史会话列表和详情浏览

#### 3.2.4 底部操作栏

- 快捷操作按钮：手动触发回答、复制转写、手动精修
- 当前模式状态指示（LLM 过滤模式等）

### 3.3 系统托盘

- 应用最小化到系统托盘（后端服务持续运行）
- 托盘图标显示当前录音状态
- 托盘右键菜单：打开浏览器、开始/停止监听、退出
- 双击托盘图标在浏览器中打开界面

### 3.4 隐蔽性设计

- 快捷键快速最小化/恢复浏览器窗口
- 深色界面降低可见度
- 通知无声音
- 浏览器标签页标题显示为普通名称（不暴露功能）
- 支持快捷键操作，减少鼠标动作

### 3.5 实时通信

- 后端通过 WebSocket 向前端推送实时数据
- 推送内容：转写片段、问题检测结果、答案生成进度、精修状态
- 前端断线后自动重连，不丢失数据

### 3.6 设置界面

通过 Web 页面内的设置面板（模态框或独立页面）配置。

#### 3.6.1 API 密钥

- 输入框以密码模式遮蔽显示
- 首次启动时引导配置
- 已配置时显示提示文字，留空则保持不变
- 加密存储

#### 3.6.2 音频设置

- 麦克风选择（列出可用设备，支持系统默认）
- 实时 ASR 模型选择（多模型可切换，需重启会话生效）
- 课程热词输入（同时应用于实时层和精修层）

#### 3.6.3 精修 ASR 设置（可选）

- 精修功能总开关（默认关闭，有额外 API 成本，需用户主动开启）
- 精修触发策略选择：课后批量（默认） / 课中定时 / 仅手动

- 单次课堂修正上限（默认 90 分钟，防止费用失控）
- 本月精修用量统计（累计音频时长）
- 精修后联动开关：
  - 问题二次检测（可选）
  - 答案自动更新（可选）

- 授课语言选择（中文/英文）

- 英文翻译开关
- 中英双语展示开关
- 数据存储路径（更改需重启应用，旧数据不自动迁移）

## 5. 非功能需求

| 维度 | 要求 |
|------|------|
| **隐蔽性** | 深色界面、快捷键操作、快速隐藏/显示、无声通知 |

| **错误恢复** | ASR 断连自动重连；精修失败静默降级；翻译失败不影响核心功能；通知服务不可用时静默降级 |
| **首次运行** | 引导用户配置必要的 API Key |

## 6. 数据组织

- 每节课的数据以「日期 + 课程名」为单位归档
- 录音文件与会话记录关联
- 结构化数据（转写、问答）持久化存储
- 支持续记场景下多个录音文件关联同一会话

### 6.1 转写片段数据

每个转写片段记录以下信息：

| 字段 | 说明 |
|------|------|

| 实时版文本 | 始终保留（作为回退） |
| 精修版文本 | 精修完成后填入 |

| 字段 | 说明 |
|------|------|

| 精修策略 | 课后 / 课中 / 手动 |

### 6.3 问题检测来源

- `auto`：自动检测
- `manual`：手动触发
- `refined`：精修后补检

## 7. 关键产品决策摘要

| 决策点 | 决策 | 理由 |
|--------|------|------|
| 精修默认开启？ | 否，默认关闭 | 有额外成本，需用户主动选择 |
| 修正是否打断用户？ | 否，完全静默替换 | 课堂场景不能有干扰 |
| 修正失败是否通知？ | 否，静默回退 | 用户无需关心精修细节 |
| 实时版是否保留？ | 是，始终保留 | 作为回退和对比基础 |
| 默认触发策略？ | 课后自动修正 | 最简单、最安全、零课中影响 |
| 二次问题检测？ | 可选，默认开启 | 核心增值点之一 |
| 答案自动更新？ | 课中自动更新，课后仅存档 | 课中更新仍有价值 |
| 导出默认版本？ | 精修版优先 | 用户导出是为了获取最好的内容 |
| 历史显示版本？ | 最优版本（精修 > 实时） | 用户无需关心版本管理 |
| 主动提问上下文？ | 始终最优版本 | 越准确越好 |
| 说话人标签冲突？ | 以高精度为准 | 高精度分离更可靠 |

## 8. 验收标准

### 8.1 核心功能验收

- [ ] 高精度修正关闭时，系统行为与无精修功能时**完全一致**，无任何副作用
- [ ] 高精度修正开启后，实时 ASR 的延迟和功能**不受影响**
- [ ] 课后修正可在后台自动完成，无需用户干预
- [ ] 修正后的文本在转写视图中静默替换，无突兀感
- [ ] 精修后补检到的遗漏问题正确添加到问题列表
- [ ] 精修失败时静默回退到实时版，用户无感知
- [ ] 历史记录和导出使用精修后的最优文本
- [ ] 续记时正确加载已精修的历史上下文
- [ ] 成本保护机制正常工作

### 8.2 用户体验验收

- [ ] 用户不需要理解"双层 ASR"概念即可正常使用
- [ ] 课堂中无任何因精修产生的通知或弹窗干扰
- [ ] 精修状态信息不显眼，仅在用户主动关注时可见
- [ ] 设置项简洁明了，默认值合理
