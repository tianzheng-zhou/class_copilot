## 一、尚未修复的代码问题

### 🟡 2.7 `AudioCapture.stop()` 可能导致 pyaudio 异常

**审计描述**: `stop()` 中先 `stop_stream()` 再 `join` 线程，read 和 close 之间的竞态可能产生 `OSError`。

**修复状态**: 未修复（审计中标注为"低风险"）

**可行性分析**:
- **难度**: 低
- **方案**: 在 `_capture_loop` 中用一个 `threading.Event` 作为退出信号替代 `stream.is_active()` 检测，先设置 event、join 线程、再 stop_stream/close。
- **风险**: 改动涉及音频采集核心循环，需测试确保不会遗漏最后几帧数据。
- **建议**: ✅ 可做，优先级低。当前 try/except 已经兜底。

---

### 🟡 5.5 `AudioStorage` 不支持多段音频管理

**审计描述**: 续记会产生多个音频文件，但数据库只存一个 `audio_path`。

**当前修复**: 已用 `|` 分隔符拼接多路径存储到 `audio_path` 字段。

**可行性分析**:
- **难度**: 中
- **当前方案已够用**: `|` 分隔符可以工作，删除会话时按 `|` 分割逐个删除即可。
- **更好方案**: 新建 `audio_files` 表（session_id + file_path），一对多关系。需要数据库迁移 v3 + 修改 `AudioStorage`、`SessionManager.delete_session`、导出逻辑。
- **建议**: 🟡 当前方案可接受，如果续记功能使用频繁再升级为独立表。

---

### 🟡 5.6 无网络状态检测和离线处理

**审计描述**: 缺少离线状态 UI 指示、断网音频缓冲、LLM 请求队列。

**可行性分析**:
- **难度**: 高
- **网络检测**: 可用 `socket.create_connection(("8.8.8.8", 53), timeout=3)` 周期检测，或依赖 ASR 断连事件推断。
- **音频缓冲**: 需要在 `_on_audio_chunk` 中判断 ASR 状态，离线时将 PCM 数据写入临时文件，重连后批量发送。问题：实时 ASR API 不支持"补录"，重连后只能从新音频开始。
- **LLM 队列**: 网络恢复后重试失败的检测/答案生成任务，需要任务队列。
- **UI 指示**: 在状态栏显示 🟢/🟡/🔴 标识，已有 `on_status_changed` 回调基础。
- **建议**: 🟡 分步实施。UI 状态指示（简单）和 LLM 重试队列（中等）优先，音频缓冲价值不大（实时 ASR 不支持补录）。

## 二、功能优化建议（💡 8.1 - 8.14）

**描述**: 利用 `QwenClient.chat_stream()` 逐字渲染答案，减少感知延迟。

**可行性**:
- **难度**: 中
- **改动范围**: `AnswerGenerator` 新增 `generate_stream()` → `SessionManager` 用流式回调替代一次性回调 → `AnswerView` 支持增量文本追加。
- **关键挑战**: 需要一个新的信号 `sig_answer_chunk(question_id, chunk_type, text)`，UI 端按 question_id 找到对应卡片并 append。
- **预估工作量**: 2-3 小时
- **建议**: ✅ **强烈推荐**，对体验提升最大。

### 💡 8.2 问题检测本地预过滤

### 💡 8.10 断网离线缓冲与恢复

**描述**: ASR 断开时缓存音频，重连后补充转写。

**可行性**:
- **难度**: 高
- **核心问题**: 实时 ASR WebSocket API **不支持离线音频补传**。重连后只能从新音频开始处理。要补录需调用离线（非实时）ASR API，这是完全不同的接口。
- **可行替代**: 断网期间的 PCM 数据已经通过 `AudioStorage` 写入了本地 MP3 文件，理论上可在网络恢复后用离线 API 转写缓冲段，但增加复杂度和成本。
- **建议**: ❌ 投入产出比低，实时 ASR 架构决定了这个功能价值有限。本地录音已经保底。

---

### 💡 8.11 多重连策略（指数退避）

**描述**: 当前固定 2 秒重连，改为指数退避。

**可行性**:
- **难度**: 低
- **改动**: `_on_asr_disconnected` 中维护 `_reconnect_attempt` 计数器，delay = min(2^attempt, 30)。
- **预估工作量**: 30 分钟
- **建议**: ✅ 简单有效，避免频繁重连浪费资源。

---

### 💡 8.12 输入法快捷键冲突检测

**描述**: 默认快捷键 `Ctrl+Shift+S/H/F` 可能与输入法冲突。

**可行性**:
- **难度**: 高（真正的冲突检测很难）
- **可行方案**: 
  - 简单方案：README 标注已知冲突 + 默认使用 `Ctrl+Alt+` 前缀避免。
  - 检测方案：尝试注册快捷键，失败则提示用户。但 `pynput` 不提供注册失败回调。
- **建议**: 🟡 改默认快捷键前缀为 `Ctrl+Alt+` 即可，不值得做检测。
